Getting Started with Citadel

Citadel is a runtime AI governance kernel. It wraps every agent action in a policy evaluation engine — returning a traceable decision with a terminal status, the exact winning_rule that determined the outcome, and a tamper-evident audit event. This guide takes you from zero to a governed action in under five minutes.

Installation

Citadel is a Python package with a FastAPI server and PostgreSQL backend. The simplest start is Docker Compose:

Shell
# Clone and start
git clone https://github.com/citadel-gov/citadel
cd citadel
cp .env.example .env  # set CITADEL_API_KEY and DATABASE_URL
docker compose up -d

# Verify
curl http://localhost:8000/health

Or install the Python package directly:

Shell
pip install citadel-governance

# Set environment variables
export CITADEL_API_KEY="sk_your_key_here:admin"
export DATABASE_URL="postgresql://user:pass@localhost/citadel"
API key format

API keys follow the format sk_value:scope1,scope2. The admin scope grants full access. For production, use scoped keys: sk_key:read or sk_key:write.

Quickstart: Your first governed action

Every agent operation is submitted to Citadel as an Action. The action_name must match the pattern namespace.verb — for example price.update, db.write, or model.execute.

1. Submit an action

POST /v1/actions
curl -X POST http://localhost:8000/v1/actions \
  -H "Authorization: Bearer sk_your_key:admin" \
  -H "Content-Type: application/json" \
  -d '{
    "actor_id":    "my-agent-001",
    "actor_type":  "agent",
    "action_name": "price.update",
    "resource":    "product/SKU-8821",
    "payload":     {"new_price": 149.99},
    "context":     {"environment": "production"},
    "dry_run":     false
  }'

2. Read the decision

Every response carries a terminal status value (lowercase). The status is your gate: if executed, the action ran; if blocked_policy, it was stopped; if pending_approval, it needs a human reviewer.

Response 202
{
  "action_id":    "3f9a2c1d-8b4e-4f2a-9c1d-5e6f7a8b9c0d",
  "status":       "executed",
  "winning_rule": "policy_allow_standard",
  "reason":       "Within policy thresholds",
  "executed":     true,
  "result":       { "updated": true },
  "error":        null
}

3. Handle all statuses

Your code should handle all terminal statuses. The most common are:

Python
import httpx

response = httpx.post(
    "http://localhost:8000/v1/actions",
    headers={"Authorization": "Bearer sk_your_key:admin"},
    json={
        "actor_id":    "my-agent-001",
        "actor_type":  "agent",
        "action_name": "price.update",
        "resource":    "product/SKU-8821",
        "payload":     {"new_price": 149.99},
    }
)
decision = response.json()

match decision["status"]:
    case "executed":
        # Action ran successfully
        result = decision["result"]

    case "pending_approval":
        # Human approval required — poll or use webhooks
        approval_id = decision["action_id"]

    case s if s.startswith("blocked_"):
        # Action blocked by policy or kill switch
        # decision["winning_rule"] names the exact rule
        raise GovernanceError(decision["reason"])

    case "allowed":
        # Approved but not yet executed (dry_run or capability token needed)
        pass

Authentication

All endpoints require an API key passed as a Bearer token:

HTTP Header
Authorization: Bearer sk_your_key_here:admin

The key format is value:scope1,scope2. Available scopes:

ScopeAccess level
adminFull access to all endpoints and actions
writeSubmit actions, approve/reject, create policies
readList and inspect actions, approvals, audit logs

Core Concepts

Actions

An Action is the fundamental request unit. Any operation an AI agent wants to perform is submitted as an Action — Citadel evaluates it, produces a Decision, and either executes the operation or stops it.

The action_name field is your primary policy handle. It must match the pattern ^\w+\.\w+$ — a namespace and a verb separated by a dot:

FieldTypeRequiredDescription
actor_idstringrequiredIdentifier for the agent submitting the action. Max 128 chars.
actor_typestringoptionalagent · workflow · service · user_proxy. Defaults to agent.
action_namestringrequiredFormat: namespace.verb — e.g. price.update, db.write, model.execute.
resourcestringrequiredThe specific resource being acted on. Max 256 chars. Used in scope matching for capability tokens.
payloadobjectoptionalArbitrary JSON payload for the action. Evaluated by policy rules.
contextobjectoptionalExecution context metadata — environment, session info, trace data.
idempotency_keystringoptionalPrevents duplicate execution. Same actor_id + idempotency_key returns the original decision.
capability_tokenstringoptionalA pre-issued gt_cap_* token to skip policy evaluation for pre-authorized actions.
dry_runbooleanoptionalIf true, evaluate policies but do not execute. Returns dry_run status.
root_decision_idstringoptionalLineage: ID of the root governance decision for this workflow chain.
parent_decision_idstringoptionalLineage: ID of the direct parent decision (for delegated actions).
trace_idstringoptionalDistributed trace ID for correlating across systems.
workflow_idstringoptionalWorkflow or job identifier for grouping related actions.

Decisions

A Decision is the terminal governance outcome for an Action. Every submitted Action produces exactly one Decision. The status field (lowercase) is the canonical outcome.

KernelStatus values

Status (wire value)MeaningExecuted?
executedAction was allowed and ran successfully✓
allowedAction was permitted but not yet executed (needs capability token or explicit execute call)—
dry_runPolicies evaluated, action not executed (dry_run: true)—
pending_approvalWaiting for human approval in the approval queue—
rejected_approvalHuman reviewer rejected the action✗
expired_approvalApproval window elapsed before a decision was made✗
blocked_policyA policy rule blocked the action✗
blocked_emergencyA kill switch is active for this scope✗
blocked_capabilityRequired capability token missing or invalid✗
blocked_schemaRequest failed schema validation✗
rate_limitedActor exceeded the configured rate limit✗
failed_executionPolicy allowed the action, but execution raised an error✗
failed_auditAction ran but audit logging failed (critical — investigate)✓
Status values are lowercase on the wire

The internal Python enum is KernelStatus.EXECUTED but the API serializes to lowercase: "executed". Always compare against lowercase strings.

Trust Bands

Every agent has a trust score between 0.0 and 1.0 that maps deterministically to one of five trust bands. Trust bands modify the policy evaluation context — they are governance signals, not authority sources. Authority is always granted by policy rules and capability tokens.

Trust band thresholds are immutable code-level constants. Changing them requires a code change and a policy version bump — this prevents silent threshold drift.

BandScore rangeAudit levelApproval bypass belowRate limit multiplier
Revoked 0.00 – 0.19 Full None (all actions blocked) 0×
Probation 0.20 – 0.39 Full None (no bypass in probation) 0.5×
Standard 0.40 – 0.59 Normal Below low risk 1×
Trusted 0.60 – 0.79 Enhanced Below medium risk 2×
Highly Trusted 0.80 – 1.00 Enhanced Below medium risk 5×

Band constraints

Each band applies default constraints that policies can make stricter but never looser:

  • Probation agents are forbidden from delegate and handoff operations. All orchestration is blocked until the band exits probation.
  • Revoked agents have max_spend_multiplier: 0 and rate_limit_multiplier: 0. All actions are effectively blocked.
  • Trusted and Highly Trusted agents receive enhanced audit logging — more scrutiny, not less — reflecting higher accountability for elevated privileges.
New agents start in Probation

Newly registered agents begin at band probation and must accumulate a trust score ≥ 0.40 over at least 3 days before automatic promotion to standard. The default probation period is 7 days.

Kill Switches

Kill switches halt execution at four scopes — from a single request to all activity across all tenants. When a kill switch is active, any action in scope returns blocked_emergency.

ScopeWhat it stopsWho can activate
requestA single specific action IDAny operator
agentAll actions from one actorAny operator
tenantAll actions within a tenantTenant admin
globalAll actions everywherePlatform admin only
Kill switches in Revoked band trigger an automatic band drop

Agents in the probation band that trigger a kill switch are automatically dropped to revoked. This is a code-level safety rule, not a configurable policy.

Audit Chain

Every governance event is appended to a cryptographic hash chain. Each AuditEvent carries an event_hash computed from its content plus the prev_hash of the preceding event. Tampering with any event breaks all subsequent hashes — detectable via GET /v1/audit/verify.

Audit event types, in order of a typical action lifecycle:

  1. action_received — Action submitted to kernel
  2. kill_switch_checked — Kill switch evaluation
  3. capability_checked — Capability token evaluation
  4. risk_assessed — Risk level and score computed
  5. policy_evaluated — Policy rules evaluated, winning rule selected
  6. approval_requested — Approval gate triggered (if applicable)
  7. approval_granted / approval_denied
  8. decision_made — Terminal decision recorded
  9. action_executed / action_failed

Multi-Agent Orchestration

When agents spawn subagents or transfer context, Citadel's orchestration API maintains a continuous governance lineage. Every child action traces back to its root authorization via root_decision_id, parent_decision_id, trace_id, and workflow_id.

The four orchestration primitives:

OperationEndpointWhat it does
delegatePOST /v1/orchestrate/delegateGrant scoped authority to a child agent. Child scope is always ≤ parent scope.
handoffPOST /v1/orchestrate/handoffTransfer active authority to a new agent atomically. Parent loses the grant.
gatherPOST /v1/orchestrate/gatherRun parallel branches under one root decision. All branches share the same lineage trace.
introspectPOST /v1/orchestrate/introspectRuntime safety check on any in-flight grant. Returns scope coverage, expiry, kill-switch status.
Probation agents cannot orchestrate

Agents in probation or revoked trust band are forbidden from delegate and handoff operations. Submit the action and it will return blocked_capability.

API Reference

Overview

All endpoints are under https://your-host/v1/. Every request requires Bearer auth. The API returns JSON; errors follow the FastAPI default exception shape with a detail field.

Base URL
https://your-citadel-host/v1

Actions

POST /v1/actions Submit action for governance · 202 Accepted

The primary governance entry point. Evaluates kill switches, capability tokens, trust bands, policy rules, and risk assessment in sequence. Returns a terminal ActionResponse.

Also available as POST /v1/actions/execute — identical behavior, explicit naming.

Request body

tenant_idstringoptionalTenant context. Inferred from API key if omitted.

idempotency_keystringoptionalIdempotency token. Max 256 chars.
capability_tokenstringoptionalPre-issued gt_cap_* token.

root_decision_idstringoptionalOrchestration lineage: root decision ID.
parent_decision_idstringoptionalOrchestration lineage: parent decision ID.

Approvals

When an action returns pending_approval, it enters the approval queue. Reviewers can approve or reject via API or the dashboard.

GET /v1/approvals List approvals · tenant-scoped

Query parameters

ApprovalResponse fields

POST /v1/approvals/{approval_id}/approve Approve a pending request
Request body
{
  "reviewed_by": "operator@corp.com",
  "reason":      "Verified within budget cap for Q2"
}

If the approval is not in pending status, returns 409 Conflict.

Policies

Policies define the governance rules evaluated by the kernel. Citadel also provides a no-code control for the most common use case — routing actions above a risk threshold to the approval queue.

POST /v1/policies/no-code/approval-threshold Apply a risk threshold policy — no YAML required

The most common governance need — route actions above a risk score to human review — with no policy authoring required.

Request body
{
  "risk_score_threshold": 70,
  "approval_priority":   "high",
  "approval_expiry_hours": 24,
  "reason":               "Route high-risk actions to compliance team"
}

Use POST /v1/policies/no-code/approval-threshold/preview first to see the generated policy without applying it.

FieldTypeConstraints
risk_score_thresholdinteger0–100. Actions with score > threshold enter the approval queue.
approval_prioritystringlow · medium · high · critical
approval_expiry_hoursinteger1–168 (1 week max). Default 24.
reasonstringOptional. Max 512 chars.

Governance Framework Mappings

EU AI Act

Citadel maps directly to the human oversight and accountability requirements of the EU AI Act for high-risk AI systems.

ArticleRequirementCitadel implementation
Art. 9Risk management systemRisk scoring (0–100) on every action; trust bands; PathType selection (fast/standard/high_risk)
Art. 14Human oversightApproval queue with mandatory human review for pending_approval decisions; kill switches accessible within one click
Art. 17Quality management and documentationImmutable policy snapshots; audit chain with event_hash verification; evidence packet export

NIST AI RMF

FunctionNIST RMF categoryCitadel feature
GovernOrganizational policies and accountabilityPolicy engine with framework tags (NIST, SOC2, GDPR…); audit attribution
MapRisk context and identificationRisk level enum (none/low/medium/high/critical) and risk score (0–100) per action
MeasureAnalysis and assessmentTrust band scoring; governance posture dashboard; audit chain integrity verification
ManageRisk response and treatmentKill switches; approval gates; capability tokens for scoped authorization; no-code threshold policies

Integrations

LangGraph

Wrap any LangGraph node or state graph with governed execution. Every node transition is submitted as a Citadel action with full lineage preserved.

Python
from citadel.integrations import GovernedNode, GovernedStateGraph

# Wrap a node — all state transitions are governed
@GovernedNode(
    action_name="langgraph.node_execute",
    resource="workflow/pricing",
)
async def pricing_node(state: dict) -> dict:
    ...

# Or wrap an entire graph
graph = GovernedStateGraph(
    StateClass,
    governance_server=LangGraphGovernanceServer(
        citadel_url="http://localhost:8000",
        api_key="sk_your_key:write",
    )
)

Claude Code

Govern Claude Code tool calls and agent sessions. Each tool invocation becomes a Citadel action.

Python
from citadel.integrations import GovernedClaudeCode, ClaudeCodeGovernanceServer

claude = GovernedClaudeCode(
    governance_server=ClaudeCodeGovernanceServer(
        citadel_url="http://localhost:8000",
        api_key="sk_your_key:write",
        actor_id="claude-code-session-001",
    )
)

# Tool calls are governed: action_name = "tool.{tool_name}"
result = await claude.run_task("Analyze the pricing data")
